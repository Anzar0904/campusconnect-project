# CampusConnect — Security Hardening

## What was fixed and why

---

### 🔴 P0 — Critical (exploitable without login)

#### 1. Server-side email domain enforcement
**Problem:** `signInWithOtp()` was called directly from the browser. Anyone could `curl` the Supabase auth endpoint with `attacker@gmail.com` and receive a magic link.

**Fix (two layers):**
- **Edge Function** `validate-otp` — the login page now calls this instead of Supabase directly. It queries the `colleges` table and only sends an OTP if `@domain` matches an active college.
- **DB trigger** `trg_enforce_college_email` — fires `BEFORE INSERT` on `profiles`. Even if someone bypasses the Edge Function and calls Supabase auth directly, the trigger raises an exception and the profile row is never created.

**To complete:** In the Supabase dashboard → Authentication → Providers, disable the raw "Email" OTP endpoint so it can only be called server-side.

**Files:**
- `supabase/functions/validate-otp/index.ts`
- `supabase/security.sql` (trigger)
- `src/app/auth/login/page.tsx` (updated to use Edge Function)

---

#### 2. College-scoped RLS policies
**Problem:** `using (auth.role() = 'authenticated')` meant any logged-in user from any college could read all profiles, posts, and messages across all colleges.

**Fix:** All major tables now use `college_id = public.my_college_id()` as their RLS predicate. `my_college_id()` is a `security definer` function — it reads the caller's `college_id` from `profiles` once and can be indexed efficiently.

**Affected tables:** `profiles`, `posts`, `messages`, `marketplace_items`, `lost_found`, `dating_profiles`, `notes`, `announcements`, `confessions`

**File:** `supabase/security.sql`

---

### 🟡 P1 — Medium (exploitable after login)

#### 3. File upload validation
**Problem:** `<input accept="...">` is a UI hint only. Anyone can remove it from DevTools and upload `.exe`, 500 MB files, or spoofed MIME types.

**Fix:** Edge Function `upload-file`:
- Reads the first 8 bytes of every file and checks magic bytes (PDF, PNG, JPEG, WebP, DOCX/ZIP). Blocks Windows PE (`.exe`, `.dll`) and ELF executables regardless of extension.
- Enforces per-bucket size limits (notes: 20 MB, marketplace images: 5 MB, avatars: 2 MB).
- Rejects any MIME type not in the bucket's allowlist.
- Rate-limits uploads to 20 per hour per user.
- Storage buckets are also hardened with `allowed_mime_types` and `file_size_limit` at the Supabase level.

**File:** `supabase/functions/upload-file/index.ts`, `supabase/storage-policies.sql`

---

#### 4. Post likes — duplicate prevention
**Problem:** The old `increment_post_likes` RPC incremented the counter without checking `post_likes`. Calling it in a loop gave unlimited likes.

**Fix:** New `toggle_post_like(p_post_id, p_user_id)` RPC:
- Checks `post_likes` table first.
- Uses `INSERT ... ON CONFLICT DO NOTHING` as a secondary guard.
- Returns `{ liked: bool, likes_count: int }` so the client stays in sync.
- `DashboardClient.tsx` updated to call the new RPC.

**File:** `supabase/security.sql`, `src/app/(app)/dashboard/DashboardClient.tsx`

---

#### 5. True anonymous confessions
**Problem:** `author_id` was stored in the `posts` table for confessions. Any Supabase dashboard admin could query and de-anonymise every confession.

**Fix:**
- Confessions moved to a separate `confessions` table with **no `author_id` column** visible to student RLS policies.
- An `_author_id_audit` column (prefixed with `_`) is populated only by the `post-confession` Edge Function using the service role — it is never returned by any student-facing SELECT policy.
- Students cannot INSERT into `confessions` directly (no insert policy exists).
- A disclaimer is shown in the UI: *"Your identity is stored in a private audit log accessible only to campus administrators."*

**File:** `supabase/functions/post-confession/index.ts`, `supabase/security.sql`, `src/app/(app)/dashboard/DashboardClient.tsx`

---

### 🟢 P2 — Low (hardening)

#### 6. DB-level rate limiting
A `rate_limit_log` table + `check_rate_limit(action, limit, window)` function provides server-side throttling for any action. Currently wired into:
- Confessions: 5 per hour
- File uploads: 20 per hour per bucket

Extend to posts/messages by calling `check_rate_limit` from relevant Edge Functions or DB triggers.

**File:** `supabase/security.sql`

#### 7. Private storage for marketplace images
Marketplace bucket is now `public: false`. Images are served via 7-day signed URLs generated by the `upload-file` Edge Function, so direct URL sharing outside the app doesn't work.

**File:** `supabase/storage-policies.sql`

## Deployment checklist

```bash
# 1. Run SQL files in order
# Supabase SQL editor:
#   1. supabase/schema.sql
#   2. supabase/security.sql
#   3. supabase/storage-policies.sql

# 2. Deploy Edge Functions
supabase functions deploy validate-otp
supabase functions deploy upload-file
supabase functions deploy post-confession

# 3. Set Edge Function secrets
supabase secrets set SITE_URL=https://your-domain.vercel.app
supabase secrets set SUPABASE_SERVICE_ROLE_KEY=<your-service-role-key>

# 4. In Supabase Dashboard → Authentication → Providers
#    Disable direct "Email" OTP so only the Edge Function can send OTPs

# 5. Verify RLS is enabled on all tables (should already be from schema.sql)
```
